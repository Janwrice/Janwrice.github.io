集合框架总览

Java 的集合框架（JCF）是 Java 标准库中一个专门用来存储和操作一组对象的、至关重要的组成部分。

Java 的集合只能存储对象，不能直接存储基本数据类型，需要装箱为包装类才可以。

Java 的集合全部位于 java.util 包下，是一套基于接口设计的统一容器体系。解决了"数组定长、功能单一"的痛点，其中有动态数组、链表、哈希表等数据结构。

分为 Collection 与 Map。Collection 为单列集合，存储独立的元素。Map 为双列集合，存储键值对。

Collection

Collection 是所有单列集合的根接口，定义了所有集合通用的方法，如 add、remove、contains、size、clear，有三个核心的子接口。

List 接口

List 接口有序，可重复，带索引。相当于 C++ STL 中的 vector。

具体的实现有 ArrayList、LinkedList。

Set 接口

具体的实现有 HashSet、TreeSet、LinkedHashSet，相当于 STL 中的 set 以及 unordered_set。

Queue 接口

队列，先进先出。具体的实现有 LinkedList、PriorityQueue。

Map

Map 是独立的根接口，存储键值对，key 不可重复。一个键对应一个值。

具体的实现有 HashMap、TreeMap、LinkedHashMap 等。

迭代器

Java 中所有 Collection 类型的集合都支持统一的迭代方式，Iterator 迭代器。

List<String> list = new ArrayList<>();
Iterator<String> it = list.iterator();

while (it.hasNext()) {
    String str = it.next();
    System.out.println(str);
}
  • hasNext() —— 判断是否还有下一个元素
  • next() —— 获取下一个元素
  • remove() —— 删除当前迭代到的元素

工具类

类似于 STL 中的 <algorithm> 库，java.util.Collections 是集合的工具类，提供大量静态方法操作集合。

  • 排序：Collections.sort(list)
  • 反转：Collections.reverse(list)
  • 查找：Collections.binarySearch(list, key)
  • 填充：Collections.fill(list, value)

针对 Android 开发

Android 中的列表控件的数据源、网络请求结果、本地数据缓存，都是基于集合实现的。

Android SDK 提供了大量适合移动端的优化容器：

  • SparseArray：key 为 int 类型，避免自动装箱。比 HashMap 更省内存。
  • ArrayMap：内存占用比 HashMap 更低。适合小数据量的键值对场景。

泛型

Java 的泛型与 C++ 的模板的核心目的一致，参数化类型，让一套逻辑复用多种数据类型，同时保留编译器类型检查。

泛型类

类定义时声明类型参数。

class Box<T> {
    private T data;
    public T getData() {
        return data;
    }
    public void setData(T data) {
        this.data = data;
    }
}

泛型方法

方法独立声明类型参数，与类的泛型无关。

public class Utils {
    public static <T> T getFirst(List<T> list) {
        return list.get(0);
    }
}

泛型接口

接口定义时声明类型参数，实现类的时候指定具体类型。

interface Callback<T> {
    void onResult(T result);
}

本质 —— 类型擦除

什么是类型擦除？Java 的泛型实现的底层原理是，编译器会在编译阶段把所有泛型类型参数擦除，替换为原始类型。

无边界的泛型参数会擦除为 Object，而有上界的泛型参数会被擦除为对应的上界类型如 Number。

针对 Android 开发

Android 开发中所有集合、回调、数据实体都依赖泛型。

如何理解协变和逆变？

有协变和逆变之分，本质上是子类和父类，在数学的集合论概念上说，子类是父类的子集。但从代码的形式上来说，子类有着比父类更多的限制，此时从形式的限制上来看，"父类是子类的子集"，为了调和这种矛盾，引入了协变和逆变。

考虑一个"动物园喂食系统"的业务场景，假设我们要设计一个方法，功能是既能统计一堆苹果的总重量，又能统计一堆香蕉的总重量。同时还能在喂食框里添加水果。

定义水果

class Fruit {
    private double weight;
    public Fruit(double weight) { this.weight = weight; }
    public double getWeight() { return weight; }
}

定义苹果和香蕉

class Apple extends Fruit {
    public Apple(double weight) { super(weight); }
    public void washApple() { System.out.println("洗苹果"); }
}

class Banana extends Fruit {
    public Banana(double weight) { super(weight); }
    public void peelBanana() { System.out.println("剥香蕉皮"); }
}

对于第一个业务需求：统计总重量

统计一个框里水果的总重量。这个方法需要使用泛型，因为水果可能是苹果，也可能是香蕉。

如果这样写会出问题：

public class ZooKeeper {
    public static double totalWeight(List<Fruit> basket) {
        double sum = 0;
        for (Fruit f : basket) {
            sum += f.getWeight();
        }
        return sum;
    }
}

因为 List<Apple> 和 List<Fruit> 是两个完全不同的类型（泛型不变性：不管原始类型之间是什么父子关系，套上泛型后，它们之间就彻底没有任何关系了）。但实际上我们知道，Apple 是 Fruit 的"子集"，这个时候看的是概念上的子集，此时可以使用协变解决。

用协变解决：

public class ZooKeeper {
    public static double totalWeight(List<? extends Fruit> basket) {
        double sum = 0;
        for (Fruit f : basket) {
            sum += f.getWeight();
        }
        return sum;
    }
}

对于第二个业务：往框里加水果

给一个框里加水果。这个方法需要泛型，因为水果可能是苹果，也可能是香蕉。

如果使用协变会出问题：

public class ZooKeeper {
    public static void addApples(List<? extends Fruit> basket, int count) {
        for (int i = 0; i < count; ++i) {
            basket.add(new Apple(1));
        }
    }
}

因为这个时候，我们要把 Apple 放到框里，虽然概念上 Apple 是 Fruit 的子集，但限制上 Apple 的属性比 Fruit 要多，因此 Apple 不能直接放入 Fruit 里。从形式的限制上来看，Fruit 此时是 Apple 的子集。

因此可以用逆变来解决：

public class ZooKeeper {
    public static void addApples(List<? super Apple> basket, int count) {
        for (int i = 0; i < count; ++i) {
            basket.add(new Apple(1));
        }
    }
}

为什么 C++ 没有这回事？

C++ 没有显式的协变和逆变本质上是因为 C++ 在编译初期就事先生成了所有类型的处理函数，保证了运行时不会出现错误。而 Java 运行时会类型擦除，无法直接区分类型，因此需要通过协变和逆变来保证逻辑上不会出现问题。

异常处理

Java 的异常体系有严格的继承结构，所有可以抛出的对象都必须继承自 java.lang.Throwable，分为两大分支：Error 和 Exception。

Error 是系统性错误，无需捕获，而 Exception 是程序级异常，可以正常处理。其中 Exception 分为受检异常以及非受检异常。C++ 中所有的异常都是"非受检异常"，编译器不会处理这些异常。而 Java 设计了双轨机制。

  • 非受检异常：RuntimeException 及其所有子类。编译器不强制要求捕获或声明，代码可以直接编译通过。这标识了代码逻辑缺陷导致的错误，比如空指针、数组越界等。这类异常本质上是代码逻辑 bug。
  • 受检异常：除 RuntimeException 以外的所有子类。编译器要强制处理，是可预期但无法通过代码完全避免的外部异常。

try-catch-finally 捕获异常

try {
    FileInputStream fis = new FileInputStream("test.txt");
} catch (FileNotFoundException e) {
    e.printStackTrace();
} finally {
    return;
}

可以写多个 catch 块捕获不同异常。要遵循子类在前父类在后的逻辑顺序。

throws 抛出异常

写在方法签名末尾，用于声明"这个方法可能会抛出某类异常"。

public void readFile(String path) throws IOException {
    FileInputStream fis = new FileInputStream(path);
}

throw 抛出异常

用于在方法内部主动抛出一个异常对象。

public void setAge(int age) {
    if (age < 0 || age > 150) {
        throw new IllegalArgumentException("年龄不合法");
    }
    this.age = age;
}

try-with-resources

专门用于解决 finally 里需要手动关流代码的问题。会自动调用 close() 释放资源。

空安全工具

Objects 空安全工具类

即 java.util.Objects，是解决空指针异常的核心工具。是 Android 编码规范强制推荐使用的类。

空对象的安全比较

// 危险写法：若str为null直接崩溃
boolean result = str.equals("test");
// 安全写法：推荐替代所有字符串/对象比较
boolean safeResult = Objects.equals(str, "test");

强制非空校验

对象为空时会直接抛出空指针异常：

public void setData(String data) {
    // 参数为null直接抛异常，避免后续隐性bug
    this.data = Objects.requireNonNull(data, "data不能为null");
}

Optional 空安全容器

是一个可以容纳 null 对象的容器类。

创建 Optional

// 值一定非空，传null直接报错
Optional<String> opt1 = Optional.of("hello");
// 值可能为空，最常用
Optional<String> opt2 = Optional.ofNullable(null);
// 空的Optional
Optional<String> opt3 = Optional.empty();
