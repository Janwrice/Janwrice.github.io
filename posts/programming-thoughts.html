编程杂谈：Lambda、协变逆变、面向对象与封装

2026-07-20

对 Lambda 的一些理解

Lambda 真正的设计理念，实际上是为了实现一个"我写了一段代码，但我不关心它具体在哪里运行，只要把它交给需要这个函数的接口就好"的功能。

换句话说，Lambda 把"行为"变成了一种可以传递的值。以前我们要传递行为，得先定义一个类、实例化、再传进去——也就是匿名内部类那一套。Lambda 把这整坨样板代码全部省掉了，只留下真正有意义的那段逻辑。

所以 Kotlin 这些语言会搞出"尾随 Lambda"这种奇怪的语法，本质上就是深刻体现了这个思想：最后一个参数如果是一个函数，那它大概率就是你给这个方法"补充"的一段逻辑，把它放在括号外面，读起来就像是你在给这个调用写一段代码块。语法形式服务于设计意图，这个语法就是在明确地表示——"嘿，我给你一段逻辑，你拿去用"。

// 尾随 Lambda：最后一个参数是函数时，可以写到括号外面
button.setOnClickListener { view ->
    // 这段逻辑交给 setOnClickListener，我不关心它何时被调用
    println("clicked")
}

Java 的函数式接口（SAM 接口）也是同样的思路——有且仅有一个抽象方法的接口，就可以用 Lambda 来简写。本质上就是要传递"一段逻辑"，而那个接口唯一的方法就是这段逻辑的载体。

如何理解协变和逆变？

协变和逆变有"协"和"逆"之分，本质上是子类和父类的关系问题。从数学的集合论角度来看，子类是父类的子集。但从代码的形式限制上来看，子类有着比父类更多的属性和方法，所以从"约束"的角度来看，父类反而是子类的子集。为了调和这两种视角的矛盾，编程语言引入了协变和逆变。

协变解决的是"概念上子类是父类的子集"的场景——我要从一堆东西里读出来用。
逆变解决的是"形式上父类的约束更少"的场景——我要往一堆东西里写进去。

考虑一个"动物园喂食系统"的业务场景。我们要设计两个功能：一是统计一堆水果的总重量，二是往框里添加水果。

定义水果类

class Fruit {
    private double weight;
    public Fruit(double weight) { this.weight = weight; }
    public double getWeight() { return weight; }
}

class Apple extends Fruit {
    public Apple(double weight) { super(weight); }
    public void washApple() { System.out.println("洗苹果"); }
}

class Banana extends Fruit {
    public Banana(double weight) { super(weight); }
    public void peelBanana() { System.out.println("剥香蕉皮"); }
}

业务一：统计总重量 —— 协变登场

需求：统计一个框里水果的总重量。这个方法需要支持泛型，因为框里的水果可能是苹果，也可能是香蕉。

直觉上我们会这样写：

public class ZooKeeper {
    public static double totalWeight(List<Fruit> basket) {
        double sum = 0;
        for (Fruit f : basket) {
            sum += f.getWeight();
        }
        return sum;
    }
}

但这段代码有问题：你传入 List<Apple> 时编译器会直接报错。

为什么？因为 Java 的泛型是不变性（Invariance）的——不管原始类型 Apple 和 Fruit 之间有怎样的父子继承关系，套上泛型之后，List<Apple> 和 List<Fruit> 就是两个完全无关的类型。编译器不认它们之间的任何关系。

但在逻辑上我们知道，Apple 是 Fruit 的子类，苹果框凭什么不能用来统计总重量？这就是从概念上的子集来看问题。此时使用协变（Covariance）来解决：

public class ZooKeeper {
    public static double totalWeight(List<? extends Fruit> basket) {
        double sum = 0;
        for (Fruit f : basket) {
            sum += f.getWeight();
        }
        return sum;
    }
}

? extends Fruit 的意思是："这个泛型参数可以是 Fruit 的任意子类"。编译器理解了这层关系，List<Apple> 就能传进去了。

但代价是什么？ 使用了 ? extends Fruit 之后，你就不能往这个 list 里添加任何元素（除了 null）。因为编译器只知道里面的东西"至少是 Fruit"，但具体是什么它不确定——往里放 Apple 不行（万一是 List<Banana> 呢？），往里放 Fruit 也不行（万一是 List<Apple> 呢？）。这个限制是合理的：你能安全地读出来，但不能写进去。

这就是著名的 PECS 原则中的 PE（Producer Extends）——如果泛型容器只用来生产/提供数据给你用，就用 extends。

业务二：往框里加水果 —— 逆变登场

需求：给一个框里添加苹果。

直觉上这样写：

public class ZooKeeper {
    public static void addApples(List<? extends Fruit> basket, int count) {
        for (int i = 0; i < count; i++) {
            basket.add(new Apple(1.0));  // 编译错误！
        }
    }
}

这行不通。原因上面说了：用了 extends 就不能往里写。

换个角度想：我们要把 Apple 放进框里。概念上 Apple 是 Fruit 的子集没错，但从形式的限制上来看，Apple 比 Fruit 多了 washApple() 等方法，约束更多；而 Fruit 的约束更少。如果此时看"谁的约束更少，谁就是子集"，那 Fruit 反而是 Apple 的"子集"——这就是从形式约束来看的父子关系反转。

此时需要用逆变（Contravariance）来解决：

public class ZooKeeper {
    public static void addApples(List<? super Apple> basket, int count) {
        for (int i = 0; i < count; i++) {
            basket.add(new Apple(1.0));
        }
    }
}

? super Apple 的意思是："这个泛型参数可以是 Apple 的任意父类"。你传 List<Fruit> 可以，传 List<Object> 也可以。把 Apple 放进去是安全的——因为无论这个 list 的元素类型是 Fruit 还是 Object，Apple 都能赋值给它。

代价呢？ 从 List<? super Apple> 里读出来的东西，编译器只能确定它是 Object（因为最坏情况下它只是 List<Object>）。你丢失了类型信息。

这就是 PECS 原则中的 CS（Consumer Super）——如果泛型容器只用来消费/接收数据，就用 super。

一句话总结

场景 关键字 能做什么 不能做什么
协变 ? extends T 从容器读数据，当做 T 来用 读 写（除 null）
逆变 ? super T 往容器写数据，写入 T 类型 写 读（只能读到 Object）

所以协变和逆变不是什么玄学概念，它本质上是编译器帮你保证类型安全的一套规则。"概念上的子集"用协变，"形式约束上的子集"用逆变——两种视角，两种泛型声明方式。

为什么 C++ 没有这回事？

C++ 没有显式的协变和逆变声明，本质是因为 C++ 的模板在编译时会为每一种具体类型组合独立生成代码（模板实例化/单态化）。vector<Apple> 和 vector<Fruit> 编译之后就完全是两个不同的类，各自有各自的机器码，运行时不存在类型模糊的问题——任何类型不匹配在编译期就已经被揪出来了。

而 Java 为了向下兼容，采用了类型擦除（Type Erasure）——泛型信息只在编译期存在，编译完就被擦掉了，运行时的 List<Apple> 和 List<Fruit> 底层都是同一个 List（原始类型）。正因为运行时无法区分具体类型，所以 Java 必须在编译期通过 extends 和 super 这些声明，来确保你不会写出运行时才会暴露出来的类型错误。

简单说：C++ 用"编译期多生成几份代码"来解决问题，Java 用"编译期多写几个声明"来解决问题。两条路，同一个目的。

对面向对象的理解

所以面向对象到底是个什么东西？

市面上的教科书喜欢说"面向过程是自下而上的思维，面向对象是自上而下的思维"，或者说"面向对象的三大特性是封装、继承、多态"。听完感觉什么都没说——这些定义对于没写过几万行代码的人来说，跟没有一样。

我直接用 Python + matplotlib 来举例子。

面向过程的写法

import matplotlib.pyplot as plt
import numpy as np

x = np.linspace(0, 10, 100)
y = np.sin(x)

# 面向过程：直接调用 plt 模块的函数
plt.figure(figsize=(10, 6))          # 创建画布（隐式的当前画布）
plt.plot(x, y, label='sin(x)')       # 在当前画布上绘图
plt.xlabel('X 轴')                   # 设置当前坐标轴标签
plt.ylabel('Y 轴')
plt.legend()
plt.grid(True)
plt.show()

你会发现它非常简单，所有代码都在为一个隐式的"当前图表"服务。

但缺点也非常明显：如果同时需要操作多个图表，根本没办法确定 plt.xlabel() 到底修饰的是哪一个。因为 plt 模块内部维护了一个全局状态——"当前图表"是哪个就是哪个。你当然可以强行用创建多个 figure 编号来区分，或者创建一个图表数组，但代码很快就会变成一坨全局状态的混战。

更让人头疼的是，实际项目中经常会出现"几个图表长得基本一样，只是颜色不同"或者"只有标题不一样"的情况。面向过程的写法下，你只能复制粘贴然后修改几行，或者写一堆参数复杂的工具函数。可以跑，但灵活性和可维护性非常差。

这种情况在真实的编程生产中到处都是：QQ 群列表、植物大战僵尸中的僵尸群、电商的商品列表……都是"有一堆类似但不完全相同的东西需要管理"。

面向对象的写法

那么很自然地会想：有没有一种范式，能让每个图表自己管理自己？于是就有了面向对象：

import matplotlib.pyplot as plt
import numpy as np

x = np.linspace(0, 10, 100)
y = np.sin(x)

# 面向对象：显式创建 Figure 和 Axes 对象
fig, ax = plt.subplots(figsize=(10, 6))  # ax 就是一个"图表对象"

ax.plot(x, y, label='sin(x)')            # 调用这个对象的方法
ax.set_xlabel('X 轴')
ax.set_ylabel('Y 轴')
ax.set_title('面向对象写法')
ax.legend()
ax.grid(True)
plt.show()

一个图表就是一个对象。如果你需要十个图表，就创建十个对象，每个管好自己的事。即使后期出现了一个"只多了一个颜色属性"的图表变体，继承一下基类、改两个方法就搞定了。而且对象可以放进任何容器里统一管理——list[Chart] 里每个元素都对自己的行为负责，遍历、批量操作都非常自然。

面向对象的本质

面向对象本质上只是一种设计思想/编程范式。 只不过如果语言提供了 class 关键字，我们可以更方便地践行这种范式。如果语言没有提供类——比如 C 语言——用结构体加函数指针，照样可以模拟出类的效果。尽管 Linux 内核完全使用 C 语言开发（暂不考虑 Rust 的引入），但它的虚拟文件系统（VFS）、设备驱动模型等设计，思想内核就是深刻面向对象的。

另外要注意，面向对象和面向过程并不是互斥的。一个对象的某个方法内部，必然还是要用面向过程的方式一步步实现。它们的分工是：面向对象负责组织程序的结构——谁管什么数据、谁提供什么行为；面向过程负责实现具体的逻辑——拿到数据之后，第一步做什么、第二步做什么。

维护一个简单的脚本，用面向过程足够了。但面对一个庞然巨物般的工程项目，面向对象所提供的那套组织方式是必需的。

对封装的理解

封装本质上和 Rust 的所有权系统一样——它是一种限制，而且是一种对你有好处的限制。

Rust 通过限制对象的所有权和借用规则，强迫程序员在写代码时就考虑清楚内存的归属和生命周期，从而在编译期消灭了 use-after-free、double-free、数据竞争等问题。

封装也是一样：它强迫程序员用一个完整的类来实现一个完整的对象设计。外部不能随意访问内部状态，只能通过你暴露出来的公开接口来交互。这种限制换来的是：

  1. 你可以随意修改内部实现，而外部代码一行都不用改。 只要接口不变，内部的改动对外部来说就是无感的。
  2. 内部状态的一致性由这个类自己保证。 外部不可能把你的数据改成非法状态，因为它根本没权限碰你的私有字段。
  3. 代码的可读性和维护边界变得清晰。 每个类就是一个自包含的模块，出问题的时候你知道去哪个类里找。

所以封装不是"把变量前面加个 private 就叫封装了"。封装是一种设计约束——有意识地决定哪些东西对外可见、哪些东西藏起来自己用。每多暴露一个 public 方法，就意味着你多承诺了一个"这个接口以后不能随便改了"的契约。
